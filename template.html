What is Modern Application Architecture ?

We will introduce principles that :

  • Tends to provide sane architectures
  • Are intended for developers and ops (and devops)
  • Limits sofware erosion
  • Are used daily in major companies, in major deployments
  • Are building blocks for Cloud deployments, and PAAS

What is it NOT about ?

We won't speak about architectural patterns (or trends) :

  • Single Page Applications
  • Micro-Services Architectures
  • Asynchronous Programming (nor Coroutines)
  • Domain Driven Design
  • Message/Event based architecture
  • NoSQL vs SQL : hint, the problem is not this, it's persistence
  • Declarative vs Imperative : my goal is "be declarative when you can", it's easier for others

Twelve Factor App
  • No environment specific in codebase
  • Git-Flow
  • Unit test

1. Codebase

One codebase tracked in revision control, many deploys
  • eg compozer
  • Declare all dependencies, completly and exactly
  • Docker file
  • Vendoring
  • Versions ; add version testing / bumping part of the process

2. Dependencies

Explicitly declare and isolate dependencies
  • All env specific variables/conf should be in environment
  • or service like etcd
  • or DNS
  • Not app config, just env

3. Config

Store config in the environment
  • SGBD, SMTP, Memcache, …
  • as attached service, not as part of your app

4. Backing Services

Treat backing services as attached resources
  • Build : transform code into artifact (compile, fetch deps, …)
    • Continuous Integration: build test, mesure
    • Plus important
  • Release : avec configuration de deployment
  • Run : execute

5. Build, Release, Run

Strictly separate build and run stages
  • stateless and share nothing
  • ideally, RO FS
  • No sticky sessions (macaroons, Json Web Token 'JOT')

6. Processes

Execute the app as one or more stateless processes
  • read expected port from con (CF 3.)
  • don't expect to be the "owner" for a system ressource like a port

7. Port Binding

Export services by port binding
  • add more processes to support more load
  • do not daemonize, no PID
  • does not exclude threads/async

8. Concurency

Scale out via the process model
  • start fast
  • stop gracefully
  • die robustly

9. Disposability

Maximize robustness with fast startup and gracefull shutdown

Minimize three gaps

  • Time : Continuous Deployment : Develop=>dev, releases=>QA, master=>dev
  • Personnel : DevOps + automation
  • Tools/envs : Vagrant + easy branch deploy

10. Dev / Prod parity

Keep development, staging, and production as similar as possible
  • Do not handle logs yourself
  • Treat application logs appart
  • stdout ideal
  • tooling for agregation (logstash, splunk, fluentd, flume, ...)

11. Logs

Treat logs as event streams
  • No "hand" work
  • same env as current processes
  • automation + declarative
  • DSL

12. Admin processes

Run admin/management tasks as one-off processes
